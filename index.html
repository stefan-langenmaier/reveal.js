What is docker

good question? it's a company, a tool to manage (create, run, deploy, delete) containers and images on different platforms (linux, mac, win), a host for these images (where you can have an account, think app store) a library to interface the linux kernel container infrastructure it's a lot of different things (one of the reasons a "competitor" started it's own library rkt)

To clarify the vocabulary let's talk about some terms

Virtualisation vs Containerisation

What is the difference? Running a different kernel with maybe a different configuration? Wider or narrower: my SYSVIPC problem for exemple with full disk-encryption LUKS and apache might be better to isolate different images together with their kernels to have _more_ reproduciable containers

Full virtualised systems all run on their on kernels. Moderen hardware has support for it. Paravirtualised systems run on their on kernel, but the kernel knows its running virtualised and is optimized for that. Containerized systems run all on the same kernel. So this kernel has a global ressource control and can do a global optimzation of the resources. Possibly big performance gains now and in the future. Competing environments can "talk" to each other. So generally a good idea to use containers? Well it depends, as you exclude your kernel from the image your behaviour can change in the future when you update your kernel or when you change your distribution. The promise of reproduceable is overrated.

Containerization differences/alternatives

What are the differences? Why do they exist? uml OpenVZ Linux VServer they are only available via patches not part of the official kernel docker lxd lxc lmctfy rocket systemd-nsspawn they are all based on cgroups, which is in the vanilla kernel. so easy upgrades in the future very important point, as all your containers depend on that kernel jails zones similair things for other oses

The problem I want to talk about here

Experimenting with a container - developpment and especially testing => Changing components on multiple levels and then create a minimal production container at the end These are two different task but they are normally related to each other

Example problem

Webserver Interface Ruby DB Redmine Sum
Combinations 3 2 3 2 5 180
Combinations - Docker (*) 3 6 18 36 180 180+36=216
Combinations - Composable (+) 3 5 8 10 15 180+15=195

Curse of dimensions the non-composable build system has about 10% more combinations this is NOT the problem, they will always be of the same magnitude the problem is between the 15 and the 36 with 15 packages prepared you can create any combination you want with 36 base images prepared you can create any last image you want, you still have to create it manually and adapted to each though it is sad because the matrix will only be sparsely populated we will NEVER try all possible combinations only the ones that seem interesting and this is not always known in advance but if the work to change/experiment/try is to high we are in a kind of vendor lock-in, the base where you are starting from decides on your tools
