Il faut que ce soit le pire titre à vie

Bonjour et bienvenu à la présentation de * Microformats * Microdata * RDFa (lite) Pardon pour le titre Mais on va avoir un aperçu sur tous ces formats dans le contexte du web sémantique et avec des exemples pour tous les sujets D'abord qu'est-ce que le web sémantique et qu'est-ce que s'est passé avec le web sémantique? Un web qui est enrichi avec des métadonnées pour permettre les machines à comprendre le web. Ça veux dire que les métadonnées sont des information supplémentaires spécifier par les humaines pour aider les ordinateurs/logiciel avec la compréhension. Depuis longtemps des efforts pour le web sémantique sont faits, mais il manque encore le vrai succès.

Raisons

Avertissement : toutes les raisons sont spéculatives

  • pas de motivation
  • difficulté technique / complexité
  • tour d'ivoire
  • spécialisation
Pourquoi on n'a pas fait le même progrès pour le web sémantique comme pour le web sociaux? Où pourquoi il n y a pas des sites sociaux avec une bonne passerelle pour le web sémantique. Tous les raisons sont spéculatives et la source est mon avis
  • les gens qui fait le travail ne peux pas profiter de ce travail. ça a changer un peu avec l'inclusion de quelques métadonnées dans les engins de recherche
  • peut-être quelques problèmes sont plus facile à résoudre dans une façon centraliser
  • la communauté au tour. c'était fait par les scientifique pour les scientifique. il semble qu'on voulais inclure seulement les machines, mais les donnés sont encore créer par les humains. en général non inclusive. souvent le problème semble comme des exercices théorétique. C'est un problème dur, on veut un base solide (théorétique) et on veux permettre facilement la création à tout le monde.
  • la seule base commune au début était XML et URL. il faut qu'on apprenne un nouveaux langage (pour la description et pour les requêtes), des outils, et des passerelles
  • ...
On autre problème est que tous ces problèmes sont entrelacé. L'ensemble a créé un problème de la poule et l'œuf. Cela a empêché de joindre la masse critique

Aperçu

Les différentes approches pour résoudre les problèmes

  • Microformats
  • Microdata
  • RDFa (lite)

Toutes les approches utilisent HTML comme base

Ils veulent éviter d'utiliser un deuxième format. Le but est de réutiliser les outils existants. Les outils pour créer, publier et stocker les information. Au même temps les informations sont à côté les informations humain. Les différences entre les approches c'est quoi est possible à emboîter et comment c'est fait. L'ordre de points est un peu l'ordre historique et c'est aussi l'ordre dans quelle on va les regarder.

Microformats

  • l'approche la plus vieille
  • réutiliser les attributs du HTML
  • adapter les vocabulaires existants

L'autorité centrale est sur : http://microformats.org/

C'est l'approche le plus vieux. Ils réutilisent les attributs standards dans le balise HTML et ils les donnent un sémantique spéciale. Pour les formats ils réutilisent les spécification existants. L'autorité central est sur : http://microformats.org/

Ci-après les nom des spécifications pour un format de métadonnée va être vocabulaires. Il y a plusieurs noms et dans la lingo de microformat c'est "profile". Les autres format utilisent par exemple schema pour decérire un type de métadonné, mais pour être consistent on va utiliser vocabulaire.

Microformats - vocabulaires

  • hCalendar - événements
  • hCard - contacts
  • rel-license - licences
  • rel-nofollow - liens vers les contenus externes
  • ...
Il y a plusieurs sur le site de Microformat, mais ce sont peut-être le plus populaire ou connu. Le vocabulaires pour hCalendar est basé sur ics et hCard sur vCard.

Microformats - modèles de conceptions

  • date-design-pattern
  • include-pattern
  • rel-design-pattern
  • ...
Un autre aspect interessant de Microformat est qu'ils fournissent de design patterns pour exprimer quelques information. C'est nécessaire parce il utilise seulement les attributs HTML et ce n'est pas possible reprèsenter tous les information natives.

Microformats - exemples

hcalendar

<span class="vevent">
  <span class="summary">
    The microformats.org site was launched
  </span>
  on <span class="dtstart">2005-06-20</span>
  <time class="dtstart" datetime="2005-06-20"> May 20th, 2005 </time>
  at the Supernova Conference
  in <span class="location">San Francisco, CA, USA</span>.
</span>
Comme déjà dit hCalendar est basé sur iCalendar. Beaucoup des applications sont capable de lire ce format (Thunderbird, Outlook, Evolution). Donc, c'est un standard bien connu. Ils ont simplifié le format pour être capable d'exprimer dans les attributs HTML. Au début on peut voir HTML normale. Peut-être avec un peu trop de white-space, mais l'affichage est claire/compréhensible pour un humaine. Maintenant on veux ajouter des métadonnées. On fait des balise extra autour des sections lesquelles on veux spécifier plus en détails. On ajout l'attribut class avec les valeurs prédéfini par le vocabulaire. L'organisation est hiérarchique. Ça donne un événement avec un sommaire, un début et un location. À la fin on peut voir aussi un exemple pour le date-design-pattern avec HTML5.

Microformats - résumé

Pour

  • l'approche la plus vieille
  • la plus simple
  • commentaires sur l'accessibilité

Contre

  • réinvention de vocabulaires
  • lent avec HTML5
  • l'usage diminue

Pour

  • C'est l'approche le plus vieux, et il a eu l'usage plus grand pour quelques temps. Et
  • le simple dans le sens que les possibilité déjà existant sont exploiter
  • il y a des commentaire sur l'accessibilité dans la spécification (pour de text reader et quelle éléments doivent être visible)

Contre

  • Ils reinventent les vocabulaire. il n y a aucun façon de intégrer l'ecosystème plus grand du web sémantique
  • Ils sont lent avec l'adoption HTML%. Il y a seulement de drafts pour le vocabulaires. Peut-être c'est seulement prudent.
  • Effectivement l'usage est diminué

Microdata

  • standard officiel de W3
  • une collaboration de
    • Google
    • Yahoo
    • Microsoft
    • Yandex
  • beaucoup de vocabulaires sous schema.org
  • support pour l'API de DOM du navigateur

C'est un standard officielle de W3, mais les seules vocabulaires viennent de schema.org Le schema.org est principalement une collaboration de ces organisation. Donc, aucun surprise que c'est dessiner pour les engins de recherche. Microdata était la façon préférée pour le RichSnippet dans les résultat de Google. Et c'est encore supporté. Il y a/ avait du support directement dans le navigateurs, mais il semble que ça va changer. Mais il y a déjà de code JavaScript qui peut replacer cet fonctionnalité. C'est dessiner pour HTML5, mais aussi seulement HTML5.

Microdata - exemples

basic syntax

<div itemscope>
  <h1 itemprop="name">Avatar</h1>
  <div itemprop="director">
    Director: James Cameron (born August 16, 1954)
  </div>
  <span itemprop="genre">Science fiction</span>
  <a href="/movies/avatar-theatrical-trailer.html" itemprop="trailer">
    Trailer
  </a>
</div>
Ici on a la description d'une film et on veux enrichi les information pour les engin de recherche. Il y a deux attributs principalement, itemscope, ça va initialiser une section avec les métadonnées. Pour les valeurs on mettre un balise avec l'attribut itemprop autour. C'est une exemple à la base, pour mettre ça dans un contexte il y a une troisième attribut.

Microdata - exemples

vocabulaires

<div itemscope itemtype="http://schema.org/Movie">
  <h1 itemprop="name">Avatar</h1>

    Director: James Cameron (born August 16, 1954)
  </div>
  <span itemprop="genre">Science fiction</span>
  <a href="/movies/avatar-theatrical-trailer.html" itemprop="trailer">
    Trailer
  </a>
</div>
Avec l'attribut itemtype et une URL on peut utiliser de vocabulaires différentes.

Microdata - exemples

emboîter

<div itemscope itemtype="http://schema.org/Movie">
  <h1 itemprop="name">Avatar</h1>
  <div itemprop="director" itemscope itemtype="http://schema.org/Person">
  Director:
    <span itemprop="name">James Cameron</span>
    (born <time datetime="1954-08-16"
        itemprop="birthDate">August 16, 1954</time>)
  </div>
  <span itemprop="genre">Science fiction</span>
  <a href="/movies/avatar-theatrical-trailer.html" itemprop="trailer">
    Trailer
  </a>
</div>

Pour la prochaine section je vais commencer par RDFa lite, parce que c'est un vrai sous-ensemble de RDFa C'est aussi un standard de W3. C'est un peu plus claire comme le Microdata, la recherche de Google soutien aussi RDFa lite, tous le vocabulaires de schema.org sont compatible, mais il y a aussi beaucoup des autres vocabulaires déjà fait pour RDF. Et c'est aussi dessiné pour HTML5, c'est la raison pourquoi ça existes seulement depuis 2012. RDFa lite contient cinq nouveaux attributs * vocab * typeof * property * resource * prefix On va les regarder dans les exemples suivants.

RDFa lite - exemples

<p vocab="http://schema.org/" typeof="Person">
   My name is
   <span property="name">Manu Sporny</span>
   and you can give me a ring via
   <span property="telephone">1-800-555-0199</span>.
   <img property="image" src="http://manu.sporny.org/images/manu.png" />
   My favorite animal is the Liger.
</p>
Ici on peut voir l'information concernant une personne. On veut ajouter les métadonnées. Les premiers deux choses ce sont les attributs pour le context. vocab ça define le default pour la base de type suivant et le type est une personne. on pourrais mettre aussi l'url absolu dans le type. Pour les valeurs, ça fonctionne avec le même principe comme pour le Microdata. mais le nom des attributs est different

RDFa lite - exemples

<p vocab="http://schema.org/"
  prefix="ov: http://open.vocab.org/terms/" resource="#manu"
  typeof="Person">
   My name is
   <span property="name">Manu Sporny</span>
   and you can give me a ring via
   <span property="telephone">1-800-555-0199</span>.
   <img property="image" src="http://manu.sporny.org/images/manu.png" />
   My favorite animal is the
   <span property="ov:preferredAnimal">Liger</span>.
</p>
Richsnippets for hugo.soucy.cc
Mais dans cette exemple on aussi une information laquelle n'est pas possible exprimer dans le schema de Person. Et avec RDFa lite, ce pas seulement possible de nicher des information mais aussi de mélanger. Pour ça on peut ajouter des autres vocabulaires, non-défaut, qui sont préfixé. Dans ce cas là c'est la open vocab avec le properties preferredAnimal. Et pour compléter j'ai ajouter la cinquième attribut ressource. Avec ça on peut nommer une section de métadonnée. C'est comme id pour les balise HTML.

RDFa

  • standard officiel de W3
  • contient RDFa lite
  • au début, seulement disponible pour XHTML
On va continuer directement avec RDFa C'est aussi un standard officiel de W3 et c'est complètement compatible avec RDFa lite. Ça veut dire que chaque markup de RDFa lite est automatiquement RDFa. Si on utilise des attributs pour RDFa le markup devient automatiquement RDFa. Ça veux dire c'est aussi facile a changer entre les deux. La différence le plus grande est dans l'histoire. Au début de RDFa c'était seulement disponible pour XHTML ou en général les langage XML comme par exemple SVG. Ça fonctionne encore aujourd'hui mais avec la dernière spécification c'est aussi défini pour HTML5.

RDFa - exemples

<p>Date:
	<span property="http://purl.org/dc/terms/created"
		content="2011-09-10">
		10th of September, 2011
	</span>
</p>
Ici on peut voir l'attribut content. Normalement ce n'est pas nécessaire parce que il y a des balise dans HTML5 qui sont pour ça.

RDFa - exemples

<p>All content on this site is licensed under
  <a property="http://creativecommons.org/ns#license"
    href="http://creativecommons.org/licenses/by/3.0/">
    a Creative Commons License
  </a>.
  ©<span property="dc:date" datatype="xsd:gYear">
    2011
  </span>
  Alice Birpemswick.
</p>
Ici on peut voir plusieurs chose un property absolu, sans type et vocab. On peut voir l'attribut datatype ou on peut mettre les type explicite pour une information. Et on peut voir un préfixe implicite. Il y a quelques préfixe qui sont prédéfini, sauf il sont défini explicitement.

RDFa - extraction

rdf translator

Au début il faut qu'on extraie les données d'une page web. Il y a déjà un service web pour ça, mais c'est aussi disponible comme logiciel libre. C'est le rdf-translator. Il fait exactement ça il est capable de transformer dans les formats différents de RDF. Comme données de référence on va utiliser la page de Hugo.

RDFa - SPARQL

SELECT *
FROM <http://rdf-translator.appspot.com/convert/detect/xml/http%3A%2F%2Fhugo.soucy.cc%2F>
WHERE {?s ?p ?o}

Demo

Maintenant on peut utiliser SPARQL. La langage a l'air un peu comme SQL et ça a aussi la même fonctionnalité à la base. Pour l'explication, tous les variables dans SPARQL commence avec un point d'interrogation. Et les information sont stocker dans un forme qui s'appelle tripel. Et les élément d'un tripel s'appellent sujet, prédicat, objet. Parce on a seulement des variables et pas de conditions ça va matcher tous les donnés. La résulte va avoir l'air un peu chaotique mais on peut déjà voir quelques informations intéressant comme les intérêts. On peut filtrer pour ceux.

RDFa - SPARQL

PREFIX foaf: <http://xmlns.com/foaf/0.1/>
SELECT *
FROM <http://rdf-translator.appspot.com/convert/detect/xml/http%3A%2F%2Fhugo.soucy.cc%2F>
WHERE {<http://hugo.soucy.cc/#hs> foaf:interest ?o}

Demo

On peut lire ça maintenant presque comme une phrase. Les objets auxquelles #hs ont des intérêts.. C'est déjà très cool, mais le point intéressant ici c'est que SPARQL est seulement le moteur de recherche, mais la BD c'est une site web. Et les records de cette BD sont les éléments sur les pages. C'est un BD qui est vraiment décentraliser. Il faut qu'on se conscientise la différence.

RDFa - SPARQL

PREFIX foaf: <http://xmlns.com/foaf/0.1/>
SELECT *
FROM <http://rdf-translator.appspot.com/convert/detect/xml/http%3A%2F%2Fhugo.soucy.cc%2F>
WHERE {
  <http://hugo.soucy.cc/#hs> foaf:interest  ?o .
  
  SERVICE <http://dbpedia.org/sparql> {
    ?o <http://dbpedia.org/ontology/abstract> ?do .
  }

  FILTER(langMatches(lang(?do), "DE"))
}

Demo

Mais il y a plus. Un truc qui s'appelle fédération. On peut faire aussi des requêtes à les autres serveur SPARQL dans une rêquete. Suivant on va utiliser le SPARQL de la dbpedia, parce que je voudrais savoir plus de ses intérêts. On commence avec le même code pour trouver les interêts, mais la variable est utilisé encore une fois comme sujet dans une demande au point SPARQL de dpedia. Et on veux avoir tous les objets dans la variable ?do. Enfin on filtre ces résultat pour une langue spécifier. Ça nous donne un petit sommaire de chaque intérêt mains dans une autre langue.
