Web 2.0

  • mash-ups
  • folksonomies
  • metadata markup/semantic web
  • social
  • ...

In the beginning of the buzzword time for Web 2.0 the following terms were of equal importance * mash-ups * folksonmies * metadata markup/semantic web * social * ... We mostly got the social part, but with the cost of losing the decentralization. Mash-ups were hyped for some time, but they not really lived up to the potential, the semantic web was buried in proprietary silos. http://de.wikipedia.org/wiki/Web_2.0

Where are we?

A lot has improved since the beginning of the Web 2.0

  • openstreet map
  • dbpedia (wikipedias semantic capability)
  • freebase
  • musicbrainz
  • ...
But I like to argue that these are still centralised pieces of information. For the here presented this might be for technical reason. And everyone has the possibility to download theses dataset. If we look at social networking sites, the informations are more locked in then ever,

Reasons

Theses are all speculative

  • ivory tower (from scientist for scientist, non inclusive)
  • no incentives (the people doing the work don't profit from their work)
  • technical difficulties
  • complexity
  • specialisation (languages, tools, queries, interfaces)
  • ...
These reasons are all highly interwoven and hard problems. There are a lot of specifications (that existed for a long time) but they often seemed more like a theoretical exercise than help. This is a hard problem, putting the ideas on a sound base vs talking to the end user about first order logic. People don really have/had a reason to do all that metadata markup, as not they were profiting from it but others. Possibly the reason why it is still hard to do mashups. Some problems are just easier solved when done centralized like mapping. All the ideas for the semantic web started from a clean slate. New formats, rdf, special tools to write them (that means not the browser), a custom query langue was created (sparql) All problems could be overcome, but the problems together prevented to reach the necessary critical mass. This created a chicken and egg problem.

Survey

about approaches to solve these problems

  • microformats
  • microdata
  • RDFa lite
  • RDFa

Their common apporach is to use HTML as basis for the embedding extra data. They want to avoid using a special format. Their goal is on reusing existing tools.

There were a lot of people who saw these problems and tried to solve the problems. Reusing already existing methods for publishing and augmenting already exisiting data. Differences is what they can embed and how.

Microformats

They are the oldest approach to embed metadata in the HTML attributes

They are thaking existing formats and specifications and simplify them to be used in HTML attributes.

The central authority is the following website: http://microformats.org/

Microformats - Vocabularies

  • hCalendar - events
  • hCard - people, organizations, contacts
  • rel-license - licensed content
  • rel-nofollow - links in untrusted 3rd party content
  • rel-tag - tag posts and pages by subject
  • ...
Vocabularies are the most general term for formats or specifications in this context so I will stick to it to not be to ambiguous. The rel-nofollow is probably still nowadays the most used markup form the microformats, we see it in a second in the examples.

Microformats - Resume

Pro

  • oldest approach
  • "simplest" approach
  • has comments on accesibility

Contra

  • reinventing the specs
  • struggling with html5
  • actual usage is dropping
  • oldest approach (had for some times the biggest usage)
  • "simplest" approach (reusing existing html attributes class could be used at the same time for styling)
  • has comments on accesibility (text reader and which elements should be visible)

Contra

  • reinventing the specs, no formal way to see it in the bigger ecosystem of the semantic web
  • struggling with html5 (no new specifications released, only drafts)
  • actual usage is dropping relative to others
  • there is microformats2 which is stable but all vocabularies are considered a draft

Global State

Set data-state="something" on a slide and "something" will be added as a class to the document element when the slide is open. This lets you apply broader style changes, like switching the background.

Custom Events

Additionally custom events can be triggered on a per slide basis by binding to the data-state name. 


Reveal.addEventListener( 'customevent', function() {
	console.log( '"customevent" has fired' );
} );

Slide Backgrounds

Set data-background="#007777" on a slide to change the full page background to the given color. All CSS color formats are supported.

Image Backgrounds

<section data-background="image.png">

Repeated Image Backgrounds

<section data-background="image.png" data-background-repeat="repeat" data-background-size="100px">

Background Transitions

Pass reveal.js the backgroundTransition: 'slide' config argument to make backgrounds slide rather than fade.

Background Transition Override

You can override background transitions per slide by using data-background-transition="slide".
