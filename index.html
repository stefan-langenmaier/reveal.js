Web 2.0

  • mash-ups
  • folksonomies
  • metadata markup/semantic web
  • social
  • distributed
  • ...

Au début de buzz au tour le mot clé "Web 2.0", les mots suivants avaient environ la même importance et on a expecté d'améliorer tous les points. Je voudrais commencer avec le point social. Ici le monde a réussi. Maintenant ma mère et capable d'envoyer un smilie quand elle veux de n'importe ou. Les mash-ups par exemples étaient très populaire pour un temps court, mais à cause de plusiers problem (interoperabilité, licenses, technologies) c'est plus une niche pour des data scientist scientist spécialisé. Il semble que c'est un nouvelle profession. Et si on regarde le point distributed on a aggraver la situation. Ajourd'hui on a des silos immense. Ici je parle des silos mechant/férme, mais aussi de silo ouvert comme wikipedia. Qu'est-ce que le semantic web? Un web qui est enrichi avec des metadonné pour permettre les machine à comprendre le web. Ça veux dire les metadonnés sont des information supplementaires spécifier par les humaines pour aider les ordinateurs/logiciel avec la comprehension. In the beginning of the buzzword time for Web 2.0 the following terms were of equal importance * mash-ups * folksonmies * metadata markup/semantic web * social * ... We mostly got the social part, but with the cost of losing the decentralization. Mash-ups were hyped for some time, but they not really lived up to the potential, the semantic web was buried in proprietary silos. http://de.wikipedia.org/wiki/Web_2.0

On est ou?

Beaucoup a changé et amélioré depuis le début de Web 2.0

  • openstreet map
  • dbpedia (wikipedias semantic capability)
  • freebase
  • musicbrainz
  • ...
C'est sont le vraiment bon sites. Ils veulent libérer et structurer les information pour tout le monde. Mais ils sont encore très centraliser. Ces sites ne sont pas les problèmes au contraire ils fournissent beaucoup des passerelles ouvert. Mais pourquoi c'est nécessaire d'avoir un grand site et pas plusieurs petit ? Pour ces sites ça a surement beaucup de raison technique. Si on regarde l'état des réseaux sociaux le monde est plus triste. But I like to argue that these are still centralised pieces of information. For the here presented this might be for technical reason. And everyone has the possibility to download theses dataset. If we look at social networking sites, the informations are more locked in then ever,

Raison

Avertissement : tous les raisons sont speculatives

  • pas de motivation
  • technical difficulties/complexity
  • ivory tower
  • specialisation
Pourquoi on n'a pas fait le même progrès pour le web semantique comme pour le web sociaux Tous les raisons sont speculatives et la source est mon avis et la source de motivation pour la présentation Raison pour travailler pour publier de metadonné sur un service
  • les gens qui fait le travail ne peux pas profiter de ce travail. ça a changer un peux avec l'inclusion de quelques metadonnés dans les engines de recherche
  • peut-être quelques problèmes sont plus facile à resoudre dans une facon centraliser
  • la communauté au tour. c'était fait par les scientist pour les scientist. il semble qu'on voulais inclurer seulement les machines, mais les donnés sont encore créer par les humains. en general non inclusive. souvent le problème semble comme des excercise theoretique. C'est un problème dur, on veut un base soldie (theoretique) et on veux permettere facilement la création à tout le monde.
  • la seule base commune au début était xml et url. il faut on apprennes de nouvelle langage (pour la description et pour les rêquetes), des outils, et des passerelles
  • ...
On autre problème est que tous ces problèmes sont entrelacé. L'ensemble a créer comme un problème de la poulet et l'oeuf. Il n'était pas possible encore de joindre la masse critique These reasons are all highly interwoven and hard problems. There are a lot of specifications (that existed for a long time) but they often seemed more like a theoretical exercise than help. This is a hard problem, putting the ideas on a sound base vs talking to the end user about first order logic. People don really have/had a reason to do all that metadata markup, as not they were profiting from it but others. Possibly the reason why it is still hard to do mashups. Some problems are just easier solved when done centralized like mapping. All the ideas for the semantic web started from a clean slate. New formats, rdf, special tools to write them (that means not the browser), a custom query langue was created (sparql) All problems could be overcome, but the problems together prevented to reach the necessary critical mass. This created a chicken and egg problem.

Tous les approche présenter utilise HTML comme base

Ils veulent eviter d'utiliser un deuxieme format. Le but est de rutiliser les outils existants. Les outils pour créer, publier et stocker les information. Au même temps les informations sont à côté les informations humain. Les differences entre les approches c'est quoi est possible à embedder et comment c'est fait. L'ordre de point est un peu l'ordre historique et c'est aussi l'ordre dans quelle on va les regarder. Their common apporach is to use HTML as basis for the embedding extra data. They want to avoid using a special format. Their goal is on reusing existing tools. There were a lot of people who saw these problems and tried to solve the problems. Reusing already existing methods for publishing and augmenting already exisiting data. Differences is what they can embed and how.

Microformats

  • l'approche le plus vieux
  • rutiliser attribus HTML
  • adapter les specification existant

L'autorité central est sur : http://microformats.org/

C'est l'approche le plus vieux. Il rutilisent les attribut standard dans le balise HTML et donne les un semantique speciale. Pour les formats ils rutilisent les specification existants. L'autorité central est sur : http://microformats.org/

They are the oldest approach to embed metadata in the HTML attributes They are taking existing formats and specifications and simplify them to be used in HTML attributes. The central authority is the following website: http://microformats.org/

Suivant les nom des specifications pour un format de metadonné va être vocabulaires. Il y a plusieurs et dans la lingo de microformat c'est "profile", mais pour être consistent on va utiliser vocabulaire

  • hCard - contacts
  • rel-license - licenses
  • rel-nofollow - liens vers les contenus externe

Il y a quelques plus sur le site Le vocabulaires pour hCalendar est basé sur ics et hCard sur vCard. The rel-nofollow est encore très populaire ajourd'hui. Vocabularies are the most general term for formats or specifications in this context so I will stick to it to not be to ambiguous. The rel-nofollow is probably still nowadays the most used markup form the microformats, we see it in a second in the examples.

In the wild: Mozilla Events (déjà dans la version microformats2)

Comme déjà dit hCalendar est basé sur iCalendar. Beaucoup des applications sont capable de lire ce format (Thunderbird, Outlook, Evolution). C'est un standard. Ils ont simplifie le format pour être capable d'expresser dans les attribus HTML. Au début on peut voir HTML normale. Peut-être avec un peu trop de white-space, mais l'affichage est claire/comprehensible pour un humaine. Maintenant on veux ajouter des metadonnées. On fait des balise extra autour les section on veux specifier plus en details. On ajout l'attribut class avec les valeurs predefini par le vocabulaire. L'organisation est hierarchique. Ça donne un evenement avec un sommaire, un début et un location. À la fin on peut voir aussi un exemple pour le date-design-pattern avec HTML5. One example is hcalendar, it is a modification of the vCalendar format which is an open, standard exchange format for calendar information. A lot of applications can read it like Thunderbird, Outlook,… but the details would be too much, but the main type in that format is vevent. So they took that format and simplified it to fit into HTML. At the end we can also see an example of the date-design-pattern

C'est un standard officielle de W3, mais les seules vocabulaires viennent de schema.org Le schema.org est principalement und collaboration de ces organisation. Donc, aucun surprise que c'est dessiner pour les engines de recherche. Microdata était la façon préférée pour le RichSnippt dans les resultat de Google. Et c'est encore supporté. Il y a/ avait du support directement dans le navigateurs, mais il semble que ça va changer. Mais il y a déjà de code JavaScript qui peut replacer cet fonctionnalité. C'est dessiner pour HTML5, mais aussi seulement HTML5. Is an offical standard of W3 (http://www.w3.org/TR/microdata/), but the only implementation of profiles/ontologies is from schema.org. This (schema.org) is moslty a collaboration from Google, Yahoo!, Microsoft and Yandex. So it's to no surprise that it's mostly geared towards search engines. And microdata was the preferred way for Google's rich snippets (it is still supported). There is also some support in the browsers. But nowadays it looks like this support is removed again. Nevertheless there is JavaScript code which adds this interface again to the dom microdatajs. designed for html5 (and only html5)

RDFa lite

  • offical standard of W3
  • simpler than microdata
  • compatible to microdata vocabularies
  • subset of RDFa
  • supported by Google
  • lots of vocabularies at schema.org
  • designed for html5
RDFa Lite consists of five simple attributes; vocab, typeof, property, resource, and prefix. RDFa 1.1 Lite is completely upwards compatible with the full set of RDFa 1.1 attributes. This means that if an author finds that RDFa Lite isn't powerful enough, transitioning to the full version of RDFa is just a matter of adding the more powerful RDFa attributes into the existing RDFa Lite markup. Historically interwoven with RDFa, RDFa is older (like microformats but was only designed for xml/xhtml) that changed only very recently june 2012
